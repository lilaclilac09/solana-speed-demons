交易的旅程：400毫秒内发生了什么

从你的钱包点击到区块链最终确认 —— 400毫秒内经过六个检查点

400毫秒

你在 Jupiter 上点击兑换。在接下来的 400 毫秒内，你的交易在 Firedancer validator 内部经过六个独立检查点，随后被永久提交到 Solana 区块链。

400毫秒大约是打一个响指所需的时间。在这个窗口内，一个 validator 接收数千笔交易，对每一个签名进行密码学验证，检查重复项，为 区块 选出最优组合，执行每个程序，并将结果写入 支持分叉 的数据库。

1
QUIC 接入

原始 UDP 数据包通过 QUIC 从网络到达。NIC tile 将其解码为交易。

2
签名验证

每笔交易的 ed25519 签名 都会被检验。签名无效的交易将被静默丢弃。

3
Dedup 去重

64 位指纹用于识别 重复 提交。只有第一份副本才能继续传递。

4
区块打包

交易按 优先费 排名。Pack tile 为下一个区块选出最优的 2400 笔交易。

5
SVM 执行

SVM（Solana 虚拟机） 运行每个程序。 计算单元 被追踪并强制执行。

6
共识提交

区块 被分发给其他 validator。一旦 2/3 以上的节点达成共识，区块即被 最终确认 。

SVM 需要什么

在 SVM 能够执行哪怕一条指令之前，它必须加载一份完整的区块链状态快照。这段来自 Firedancer 测试框架的代码，展示了该快照所包含的确切内容。

CODE 

fn build_mock_bank(f: &TransactionFixture) -> MockBank {
    let bank = MockBank::default();
    {
        let mut accounts = bank.accounts.write().unwrap();

        for fa in &f.pre_accounts {
            accounts.insert(fa.pubkey,
                fixture_account_to_shared(fa));
        }

        for sv in f.sysvars.iter() {
            let mut acct = AccountSharedData::default();
            acct.set_data_from_slice(&sv.data);
            acct.set_owner(solana_sdk_ids::sysvar::id());
            acct.set_lamports(1);
            accounts.insert(sv.sysvar_id, acct);
        }
    }
    bank
}
通俗解释

创建一个空的内存"bank" —— 即 SVM 将要读取的 Solana 区块链状态模拟环境。

对于交易将涉及的每个账户：将其当前余额、所有者程序和存储数据加载到 bank 中。 公钥 用作查找索引。

加载所有 Solana 系统变量（ sysvars ）：当前租金价格、epoch 信息、最近 区块哈希 —— 以及运行时所需的 lamports 和所有权元数据。

返回填充完毕的 bank。SVM 现在拥有执行本笔交易所需的完整状态快照。

💡
零拷贝架构

在生产环境的 Firedancer 中，各 tile 之间从不复制交易数据。NIC tile 将交易字节一次性写入共享内存区域（dcache）。每个下游 tile 通过指针从同一内存位置读取数据。正因如此，Firedancer 能够每秒处理数十万笔交易 —— 内存带宽绝不浪费在数据复制上。

什么会导致交易失败

大多数失败的交易不会产生你所看到的错误 —— 它们只是从未到达目的地。了解是哪个检查点终止了交易，是调试的关键所在。

QUIC Ingest 数据包格式错误、网络格式不符，或该 validator 的 IP 被封锁。交易从未突破网络层。
SigVerify ed25519 签名 无效 —— 交易已损坏或使用了错误的密钥签名。立即丢弃，不收取手续费。
Dedup 这笔完全相同的交易已被见过 —— 是 validator 在数毫秒前已处理（或丢弃）的某笔交易的重发。钱包会不断重发；大多数重发都在此处被静默丢弃。
Pack slot 已满，或你的手续费太低 —— 该交易未被选入本次区块 slot。它可能在下一个 slot 中被重试。
Execute 程序耗尽了 计算单元 ，触及账户余额限制，或返回了错误代码。所有状态更改被回滚，但手续费仍然收取。
