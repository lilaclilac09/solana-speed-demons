Samba — 网线速度下的 MEV

Harmonic Finance 如何修改 Firedancer，在验证节点层面实现有保证的交易排序

什么是 MEV？

交易排序中隐藏着利润。如果你能保证自己的 套利 交易恰好落在一笔大额价格冲击交换之后、其他人做出反应之前，那这个价差就归你所有。这就是 MEV — Maximum Extractable Value。

MEV 并非漏洞或攻击——它是任何由单一参与方（即 出块者 ）决定交易顺序的区块链上客观存在的经济现实。在 Solana 上，每个区块的生产 validator 决定哪些交易进入区块以及进入的顺序。

🔄

套利

在 DEX A 以低价买入代币，在同一区块内、价格均衡之前，在 DEX B 以高价卖出。

🥪

三明治攻击

在大额交换之前插入买单（ 抢先交易 ），等大额交换推动价格上涨后立即卖出。那笔大额交换就是三明治的"馅料"。

💧

清算

当借款人的抵押品价值跌破阈值时，争先成为第一个 清算人 并领取奖励。

📦

捆绑包

一组交易以 原子方式 保证按特定顺序落块——否则全部失败。用于安全地实施上述各类策略。

ℹ️
MEV 本身是中性的

MEV 可以提升市场效率（套利），也可以从普通用户身上提取价值（三明治攻击）。关键在于 validator 层面是否使这些保证成为可能。Samba 让这一切成为现实。

从 Firedancer 到 Samba

Samba 是 fork 自 Firedancer 的——Harmonic Finance 在 Jump Crypto 的 validator 之上叠加了 MEV bundle 基础设施。下方的 git log 展示了 Samba 与 Firedancer 分叉的关键提交记录。

中文释义

v3.1.14 rebase = Harmonic 定期将最新的 Firedancer rebase 更新同步过来，使 Samba 始终跟上最新的性能改进。

harmonic buffer = Samba 新增了一个内部队列，在 bundle 到达 pack tile 之前先暂存进来。同时清理用于排序的 arrival_ns 时间戳。

reduce buffer time = 调整 bundle 在队列中等待的时长——等待时间过长会导致区块生产完毕前没来得及将其纳入。

pack: fix harmonic rejection condition = 精确定义何时拒绝一个 bundle：它能否放入剩余区块空间？目标 slot 是否距当前太远？

bundle: fix reset submit_leader_window_info_wait = bundle 只能在该 validator 即将成为 leader （出块者）时才能提交。此提交修复了该检查的时间窗口逻辑。

initial streaming impl = bundle 现在可以以实时流的方式提交，而非仅支持批量提交。这降低了每一毫秒都至关重要的高频交易者的延迟。

only append if not upcoming leader = 如果该节点即将不是出块者，则不再向队列追加 bundle——避免在无法使用的工作上浪费处理资源。

account for block txns in bundle/txn insert = 在选择 bundle 时，将区块中已选定的普通交易计入剩余空间的计算，确保数据准确。

harmonic fd client id = Samba 在网络上以独特标识符公告自身，让生态系统知道该节点支持 bundle 提交。

为什么要修改 Validator？

为何不在应用层构建 MEV 工具——就像 Solana 上现有的解决方案那样？因为应用层的解决方案只能请求排序，只有 validator 本身才能保证排序。

💡
Validator 是唯一的真相来源

通过外部服务提交的 bundle，如果 validator 不明确遵守，仍可能被 抢先交易 。Samba 将 bundle 保证直接内嵌到 pack tile 本身，让排序由出块者在密码学层面予以保证。

以下是 Samba 与 Solana 上其他 MEV 排序方案的对比：

Samba bundle 在 validator 层面保证——pack tile 在调度任何其他交易之前先为 bundle 锁定位置
Jito（应用层） Validator 小费激励排序，但并非密码学锁定——恶意 validator 仍可重新排序
普通优先手续费 较高手续费可提高落块概率，但无法保证任何事——支付相同手续费的另一笔交易仍可能先行落块
