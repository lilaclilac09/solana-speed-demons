什么是 Validator？

每一笔 Jupiter 兑换、每一次 NFT 铸造、每一次 DeFi 清算——所有这些都要经过一台叫做 validator 的机器。Validator 接收来自所有用户的全部 transactions， 对它们排序、执行，并将结果写入链上。 每 400 毫秒执行一次。

这 400ms 的窗口叫做 slot。 在每个 slot 中，一个被指定的领导者必须收集数千笔交易、验证每一个密码学签名、执行所有程序，并写出一个 block ——然后下一个 slot 就开始了。下面展示这段旅程的全貌：

规模有多大

Solana 的目标是每秒约 5,000 笔交易。这意味着 validator 必须大约每 0.2 毫秒处理一笔交易——比相机快门眨眼还快。这正是为什么 validator 工程与普通 Web 开发截然不同。

三大项目

Solana 最初只有一个 validator client： Agave（前身为 Solana Labs）。三支团队审视这一现状后，认为现有方案还不够好。

🔥

Firedancer

Jump Crypto 用 C 语言从零重建了整个 validator，目标是每秒 100 万笔交易——这是完整 Solana 协议的第二个独立实现。

🎵

Samba

Harmonic Finance 在 Firedancer 基础上分叉，并加入了 MEV 捆绑基础设施——为支付优先费用的交易者提供有保障的交易排序。

⏱️

Delorean

Temporal XYZ 构建了一项服务，将每笔交易捕获为可回放的快照，让你能够调试历史——精确回放任意一笔链上交易。

💡
为什么需要多个实现？

当两个独立代码库运行同一协议时，漏洞会以分歧的形式浮出水面，而不是悄无声息地破坏数据。工程师称之为 "consensus diversity" ——这是区块链所能拥有的最重要的安全属性之一。

  enabled features: 221
  pre-accounts    : 25
  post-accounts   : 25
  expected CUs    : 104291

--- replay outcome ---
  status          : MATCH
  CUs             : MATCH
  post-state      : MATCH
通俗解释

正在连接 Temporal 的 Penrose 服务，并根据交易 ID 请求该 fixture 的历史快照。

slot 420195494 = 这笔交易发生在第 4.2 亿个区块。Solana 每年约产生 1.6 亿个区块，因此这是一笔近期交易。

enabled features: 221 = Solana 有数百个随时间激活的协议开关； 这里记录的是该交易执行时刻哪些开关处于激活状态。

pre-accounts: 25 = 交易执行前读取了 25 个钱包和程序账户——这些是输入数据。

expected CUs: 104,291 = 原始 CU 计数来自最初的 validator 运行。本地回放必须与此完全一致。

三项全部 MATCH = 使用 SVM 进行的本地回放与原始 validator 产生了完全相同的结果。快照完美无误。
