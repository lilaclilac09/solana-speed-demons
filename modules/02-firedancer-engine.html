Firedancer：C 语言打造的极速引擎

Jump Crypto 如何借鉴芯片制造工程理念、从零构建 validator：一个 CPU 核心，一项任务，零共享工具。

为什么选择 C？

Jump Crypto 使用与其 高频交易 系统相同的工程理念编写了 Firedancer： 一切用 C 编写，掌控每一个字节，绝不留有偶然。

⚡

无垃圾回收器

垃圾回收器 会在不可预知的时刻暂停程序执行。 在 HFT 场景下，哪怕一毫秒的暂停也意味着真实的金钱损失。 C 没有垃圾回收器——你精确掌控内存。

🎯

可预测的性能

Rust 提供内存安全保证，但其抽象层可能引入隐性开销。 C 编译后与你所写的代码一一对应——每次都达到纳秒级的可预测性。

🔍
独立押注

Firedancer 完全从零编写，与现有的 Agave validator 没有任何共享代码（Agave 用 Rust 编写）。这意味着 Agave 实现中的 bug 不会影响 Firedancer——反之亦然。 这就是"客户端多样性"的本质定义。

Tile：一个核心，一项任务

tile 是绑定到单个 CPU 核心 上的单线程，只做一件事。 NIC tile 负责接收数据包。 SigVerify tile 负责验证签名。 Pack tile 负责筛选哪些交易进入区块。 各 tile 之间从不直接共享内存——如同芯片制造流水线上的各个工位。

src/ Firedancer 源码根目录
tango/ 神经系统。所有 tile 通过环形缓冲区在此通信。
ballet/ 数学库。SHA-256、ed25519、base58——全部从零实现。
disco/ 通用流水线 tile：QUIC、SigVerify、Dedup、Pack。
discof/ 完整 Firedancer tile：共识、RPC、回放，无 Agave 依赖。
discoh/ Frankendancer tile：通往 Agave 执行层的桥接路径（混合模式）。
flamenco/ Solana 运行时。程序在 SVM 上实际执行的地方。
funk/ 账户数据库：支持分叉感知的内存键值存储。
waltz/ QUIC 之下的网络层——原始 UDP 与内核旁路。
wiredancer/ FPGA 加速模块——未来的硬件加速路径。
util/ C 环境基础设施：内存分配器、数据结构、数学原语。
💡
芯片制造工厂的比喻

把每个 tile 想象成半导体流水线上的专属工位。 没有工位会借用另一工位的工具，没有工位需要等待另一工位完成。 工件在传送带（即 环形缓冲区）上到来，完成处理，继续传递——速度与网线一致。

Tango：Tile 的神经系统

各 tile 之间从不通过相互调用来传递数据。 NIC tile 将原始交易字节写入共享内存区域，并更新 环形缓冲区 条目。 下游 tile 从同一块内存中读取数据——无需拷贝，无需 锁， 无需等待。 这是为纳秒级吞吐量而设计的 IPC。

🗂️

mcache

元数据缓存。 一个循环的"信封"环——每条条目描述一笔交易（其大小、在内存中的位置以及哈希值），而不包含数据本身。 写者在每个信封上盖章；读者扫描环形结构寻找新的印记。

📦

dcache

数据缓存。 实际的交易字节。一次性存储在共享内存中。 所有 tile 从同一位置读取——永远不会产生拷贝。 就像一块公告板，所有人都能阅读，无需各自复印一份。

🎟️

fseq

流序列器。 一个取号计数器。即使多个 NIC tile 同时写入，也能确保各 tile 按正确顺序读取消息。 就像熟食柜台的"当前服务号"显示屏。

🚦

fctl

流量控制。 一个背压信号。如果下游 tile 处理速度跟不上，fctl 会通知上游 tile 降速，而非溢出环形缓冲区。 就像高速公路拥堵时拦截来车的交通灯。

Frankendancer 与完整版 Firedancer

Firedancer 从同一代码库中衍生出两个 validator。 Frankendancer 已在 主网 上线。 完整版 Firedancer 仍在开发中。
