Delorean — A Transaction Time Machine

How Temporal's Penrose service captures a complete snapshot of every transaction as blocks are produced — and lets you replay history with surgical precision

The debugging problem

🚨
Scenario

A trading bot lost $40,000 on a transaction that should have succeeded. The transaction is buried in a Solana block from 3 days ago. The network has moved on. The blockchain state at that exact moment no longer exists on any running node. How do you debug this?

Standard blockchain debugging gives you almost nothing. You can see the transaction failed, but to understand why, you need to know the exact state of every account, every program, and every system variable at that moment in time. And that state is gone.

Temporal XYZ built Penrose — a service that runs alongside Solana validators, capturing a full state snapshot for every transaction at the moment its block is produced. Delorean is the tool that fetches those snapshots and replays them.

💡
Why this is possible

Temporal runs infrastructure that captures a transaction fixture for each transaction as the validator produces the block. Each fixture contains everything the Solana VM needs to replay the transaction: pre and post account state, programs, sysvars, active feature flags. It's a perfect time capsule.

What's in a fixture?

Each fixture is a 12-field structured record — together they form a complete Solana-universe reconstruction for one transaction.

CODE 

pub struct TransactionFixture {
    pub schema_version: u32,
    pub slot: u64,
    pub recent_blockhash: Hash,
    pub signature: Signature,
    pub enabled_features: Vec<Pubkey>,
    pub lamports_per_signature: u64,
    pub transaction: Vec<u8>,
    pub pre_accounts: Vec<FixtureAccount>,
    pub programs: Vec<FixtureProgramData>,
    pub sysvars: Arc<Vec<FixtureSysvar>>,
    pub result: ExecutionResult,
    pub post_accounts: Vec<FixtureAccount>,
}
PLAIN ENGLISH

schema_version — Format version. Supports backward compatibility as Penrose evolves.

slot — The exact Solana block number where the event happened. Each block has a precise number.

recent_blockhash — A timestamp-like value that prevents transaction replay attacks.

signature — The transaction's unique ID. This is what you pass to delorean.

enabled_features — The list of feature flags active at that moment. Solana has hundreds of protocol toggles (called features) that activate over time. This field records which were active at that exact block.

lamports_per_signature — The cost per lamport fee at that moment. Fees change over time.

transaction — Raw transaction bytes: instructions, accounts, signatures.

pre_accounts — Every account state before the transaction: each wallet balance, token account, and program account as they stood before execution.

programs — The bytecode of every smart contract touched: each relevant program's bytecode (ELF binary), captured as it existed at that exact slot.

sysvars — System variables (like current rent price): Solana system variables — rent price, epoch info, clock. Crucial for accurate replay.

result — What the original validator recorded: status, consumed compute units, return data.

post_accounts — Every account state after the transaction: each account's state after execution. Used to verify replay accuracy.

With all 12 fields, you can rebuild the exact Solana universe that existed when that transaction ran — and replay it anywhere, anytime.

The full workflow

From "transaction failed" to "fix verified against real production state", in five steps.

1
Find the transaction

Get the transaction signature from your block explorer or logs. This is the unique ID of the failed event you want to debug.

2
Fetch the fixture

Run: delorean <signature> <penrose-url>. Delorean calls getTransactionFixture and downloads the full historical snapshot.

3
Replay and compare

Delorean builds the MockBank, runs through the SVM, and prints: status MATCH/MISMATCH, compute units MATCH/MISMATCH, post-state MATCH/MISMATCH.

4
Swap and retest

Add --replace-program=<addr>:<your-fix.so> and re-run, replaying with your patched program against the same historical state.

5
Verify and deploy

If your fix makes the replay pass, you've tested against real production state. Deploy with confidence.
