The Validator: The Blockchain's Engine

What Solana's core actually is — and how three teams are competing to make it faster

What is a validator?

Every Jupiter swap, every NFT mint, every DeFi liquidation — they all flow through a machine called a validator. Validators receive all transactions from every user, order them, execute them, and write the results on-chain. Once every 400 milliseconds.

That 400ms window is called a slot. During each slot, a chosen leader must collect thousands of transactions, verify every cryptographic signature, execute all programs, and write out a block — then the next slot begins. Here's the full journey at a glance:

The scale of it

Solana targets around 5,000 transactions per second. That means the validator has to process one transaction every ~0.2 milliseconds — faster than a camera shutter blinks. Which is why validator engineering looks nothing like ordinary web development.

The three projects

Solana originally had just one validator client: Agave (formerly Solana Labs). Three teams looked at this and decided the status quo wasn't good enough.

🔥

Firedancer

Jump Crypto rebuilt the entire validator from scratch in C, targeting 1 million transactions per second — the second independent implementation of the full Solana protocol.

🎵

Samba

Harmonic Finance forked Firedancer and added MEV bundle infrastructure — guaranteed transaction ordering for traders who pay priority fees.

⏱️

Delorean

Temporal XYZ built a service that captures every transaction as a replayable snapshot, letting you debug history — replay any historical on-chain transaction exactly.

💡
Why multiple implementations?

When two independent codebases run the same protocol, bugs surface as disagreements rather than silent data corruption. Engineers call this "consensus diversity" — one of the most important safety properties a blockchain can have.

Firedancer: A Wire-Speed Engine in C

How Jump Crypto rebuilt the validator from scratch, borrowing semiconductor-fab thinking: one CPU core, one job, zero shared tools.

Why C?

Jump Crypto wrote Firedancer using the same engineering philosophy as their high-frequency trading systems: everything in C, control over every byte, nothing left to chance.

⚡

No garbage collector

A garbage collector pauses your program at unpredictable moments. In HFT, even a millisecond pause means real money lost. C has no GC — you control memory precisely.

🎯

Predictable performance

Rust gives memory-safety guarantees, but its abstractions can introduce hidden overhead. C compiles to exactly what you wrote — nanosecond-level predictability, every time.

🔍
An independent bet

Firedancer is written from scratch and shares no code with the existing Agave validator (Agave is written in Rust). That means a bug in Agave's implementation can't affect Firedancer — and vice versa. This is "client diversity" by definition.

Tiles: one core, one job

A tile is a single thread pinned to one CPU core, doing one thing. The NIC tile receives packets. The SigVerify tile checks signatures. The Pack tile picks which transactions go into the block. Tiles never share memory directly with each other — like stations on a semiconductor-fab assembly line.

src/ Firedancer source root

disco/ Common pipeline tiles: QUIC, SigVerify, Dedup, Pack.
discof/ Full Firedancer tiles: consensus, RPC, replay — no Agave dependency.
discoh/ Frankendancer tiles: bridge path into Agave's execution layer (hybrid mode).
flamenco/ The Solana runtime. Where programs actually execute on the SVM.

waltz/ Network layer beneath QUIC — raw UDP and kernel bypass.
wiredancer/ FPGA acceleration module — the future hardware-accel path.
util/ C-environment foundation: memory allocators, data structures, math primitives.
💡
The semiconductor-fab analogy

Think of each tile as a dedicated station on a semiconductor fab line. No station borrows another's tools; no station has to wait for another to finish. Work pieces arrive on the conveyor belt (the ring buffer), get processed, and pass on — at wire speed.

Tango: the nervous system between tiles

Tiles never pass data by calling each other. The NIC tile writes raw transaction bytes into a shared memory region and updates a ring buffer entry. Downstream tiles read from the same memory — no copy, no lock, no waiting. This is IPC designed for nanosecond throughput.

🗂️

mcache

Metadata cache. A circular ring of "envelopes" — each entry describes a transaction (its size, where it lives in memory, its hash) but not the data itself. The writer stamps each envelope; readers scan the ring for new stamps.

📦

dcache

Data cache. The actual transaction bytes. Stored once in shared memory. All tiles read from the same location — no copies are ever made. Like a bulletin board: everyone reads it, nobody needs their own photocopy.

🎟️

fseq

Flow sequencer. A take-a-number counter. Even when multiple NIC tiles write at once, tiles read messages in the right order. Like the "now serving" sign at a deli counter.

🚦

fctl

Flow control. A backpressure signal. If downstream tiles can't keep up, fctl tells upstream tiles to slow down instead of overflowing the ring buffer. Like a traffic light stopping cars when the highway jams.

A Transaction's Journey: What Happens in 400 Milliseconds

From the click in your wallet to final on-chain confirmation — six checkpoints, in 400 ms

400 milliseconds

You hit "swap" on Jupiter. In the next 400 milliseconds your transaction passes through six independent checkpoints inside a Firedancer validator, and then gets committed permanently to the Solana blockchain.

400 milliseconds is about the time it takes to snap your fingers. In that window, a validator receives thousands of transactions, cryptographically verifies every signature, checks for duplicates, picks the best mix for the block, executes each program, and writes results into a fork-aware database.

1
QUIC ingress

Raw UDP packets arrive over QUIC from the network. The NIC tile decodes them into transactions.

2
Signature verification

Every transaction's ed25519 signature is checked. Transactions with invalid signatures are silently dropped.

3
Dedup

64-bit fingerprints identify duplicate submissions. Only the first copy continues.

4
Block packing

Transactions are ranked by priority fee. The Pack tile picks the top ~2,400 transactions for the next block.

5
SVM execution

The SVM (Solana Virtual Machine) runs each program. Compute units are tracked and enforced.

6
Consensus commit

The block is distributed to other validators. Once 2/3+ of nodes reach consensus, the block is finalized.

What the SVM needs

Before the SVM can execute a single instruction, it has to load a complete snapshot of the blockchain state. This snippet, from the Firedancer test framework, shows exactly what that snapshot contains.

CODE 

fn build_mock_bank(f: &TransactionFixture) -> MockBank {
    let bank = MockBank::default();
    {
        let mut accounts = bank.accounts.write().unwrap();

        for fa in &f.pre_accounts {
            accounts.insert(fa.pubkey,
                fixture_account_to_shared(fa));
        }

        for sv in f.sysvars.iter() {
            let mut acct = AccountSharedData::default();
            acct.set_data_from_slice(&sv.data);
            acct.set_owner(solana_sdk_ids::sysvar::id());
            acct.set_lamports(1);
            accounts.insert(sv.sysvar_id, acct);
        }
    }
    bank
}
In plain English

Create an empty in-memory "bank" — the simulated Solana blockchain state the SVM will read from.

For every account the transaction will touch: load its current balance, owner program, and stored data into the bank. The public key is used as the lookup index.

Load every Solana system variable ( sysvars ): current rent price, epoch info, recent blockhash — along with the lamports and ownership metadata the runtime requires.

Return the populated bank. The SVM now has the complete state snapshot it needs to execute this transaction.

💡
Zero-copy architecture

In production Firedancer, tiles never copy transaction data between each other. The NIC tile writes transaction bytes once into a shared memory region (dcache). Every downstream tile reads from that same memory through pointers. That's how Firedancer hits hundreds of thousands of transactions per second — no memory bandwidth wasted on data copies.

What kills a transaction

Most failed transactions don't produce the error you'd expect — they simply never arrive. Knowing which checkpoint killed yours is the key to debugging.

QUIC Ingest Packet malformed, wrong network format, or this validator's IP is blocking yours. The transaction never made it past the network layer.
SigVerify The ed25519 signature is invalid — the transaction was corrupted or signed with the wrong key. Dropped immediately, no fee charged.
Dedup This exact transaction has been seen before — it's a retry of one the validator already processed (or dropped) milliseconds earlier. Wallets retransmit constantly; most retries get silently dropped here.
Pack The slot is full or your fee is too low — the transaction didn't make the cut for this block slot. It may get retried in the next slot.
Execute The program ran out of compute units, hit an account-balance limit, or returned an error code. All state changes are rolled back, but the fee is still charged.

Samba — MEV at Wire Speed

How Harmonic Finance modified Firedancer to deliver guaranteed transaction ordering at the validator layer

What is MEV?

There's profit hiding in transaction ordering. If you can guarantee that your arbitrage transaction lands right after a big price-moving swap and before anyone else can react, that spread is yours. That's MEV — Maximum Extractable Value.

MEV is not a bug or an attack — it's an objective economic reality on any blockchain where a single party (the block producer ) decides transaction order. On Solana, the validator producing each block decides which transactions go in and in what order.

🔄

Arbitrage

Buy a token cheap on DEX A and sell expensive on DEX B in the same block, before prices equalize.

🥪

Sandwich attack

Insert a buy order before a big swap ( front-running ), then sell right after the big swap pushes the price up. The big swap is the "filling" in the sandwich.

💧

Liquidation

When a borrower's collateral drops below threshold, race to be the first liquidator and claim the reward.

📦

Bundles

A group of transactions guaranteed to land in a specific order atomically — or all fail. The building block for safely executing all the strategies above.

ℹ️
MEV is neutral by itself

MEV can improve market efficiency (arbitrage), or it can extract value from regular users (sandwich attacks). The point is whether the validator layer makes these guarantees possible at all. Samba is what makes that real.

From Firedancer to Samba

In plain English

v3.1.14 rebase = Harmonic periodically rebases onto the latest Firedancer release, keeping Samba up to date with upstream performance improvements.

harmonic buffer = Samba added an internal queue to hold bundles before they hit the pack tile. It also cleans up the arrival_ns timestamp used for ordering.

reduce buffer time = Tunes how long a bundle waits in the queue — too long and the block can finish without including it.

pack: fix harmonic rejection condition = Defines exactly when to reject a bundle: does it fit in the remaining block space? Is the target slot too far away?

bundle: fix reset submit_leader_window_info_wait = Bundles can only be submitted when this validator is about to be the leader (block producer). This commit fixes the timing-window logic for that check.

initial streaming impl = Bundles can now be submitted as a live stream, not just in batches. Cuts latency for HFT traders where every millisecond matters.

only append if not upcoming leader = If this node isn't about to be the producer, don't bother appending bundles to the queue — avoid wasting processing on work it can't use.

account for block txns in bundle/txn insert = When picking bundles, the remaining-space calculation now counts the regular transactions already selected for the block — keeps the accounting accurate.

harmonic fd client id = Samba advertises itself with a unique identifier on the network, so the ecosystem knows this node supports bundle submissions.

Why modify the validator?

Why not build MEV tooling at the application layer, like existing solutions on Solana? Because an application-layer solution can only request ordering — only the validator itself can guarantee it.

💡
The validator is the only source of truth

A bundle submitted through an external service can still be front-run if the validator doesn't explicitly honor it. Samba bakes the bundle guarantee directly into the pack tile itself — ordering becomes a cryptographic promise from the block producer.

Here's how Samba compares to other MEV ordering solutions on Solana:

Samba bundles Guaranteed at the validator layer — the pack tile reserves slots for the bundle before scheduling any other transaction
Jito (application-layer) Validator tips incentivize ordering, but it's not cryptographically locked — a malicious validator can still reorder
Plain priority fees Higher fees improve the odds of landing, but guarantee nothing — another transaction with the same fee can still land first

Delorean — A Transaction Time Machine

How Temporal's Penrose service captures a complete snapshot of every transaction as blocks are produced — and lets you replay history with surgical precision

The debugging problem

🚨
Scenario

A trading bot lost $40,000 on a transaction that should have succeeded. The transaction is buried in a Solana block from 3 days ago. The network has moved on. The blockchain state at that exact moment no longer exists on any running node. How do you debug this?

Standard blockchain debugging gives you almost nothing. You can see the transaction failed, but to understand why, you need to know the exact state of every account, every program, and every system variable at that moment in time. And that state is gone.

Temporal XYZ built Penrose — a service that runs alongside Solana validators, capturing a full state snapshot for every transaction at the moment its block is produced. Delorean is the tool that fetches those snapshots and replays them.

💡
Why this is possible

Temporal runs infrastructure that captures a transaction fixture for each transaction as the validator produces the block. Each fixture contains everything the Solana VM needs to replay the transaction: pre and post account state, programs, sysvars, active feature flags. It's a perfect time capsule.

What's in a fixture?

Each fixture is a 12-field structured record — together they form a complete Solana-universe reconstruction for one transaction.

CODE 

pub struct TransactionFixture {
    pub schema_version: u32,
    pub slot: u64,
    pub recent_blockhash: Hash,
    pub signature: Signature,
    pub enabled_features: Vec<Pubkey>,
    pub lamports_per_signature: u64,
    pub transaction: Vec<u8>,
    pub pre_accounts: Vec<FixtureAccount>,
    pub programs: Vec<FixtureProgramData>,
    pub sysvars: Arc<Vec<FixtureSysvar>>,
    pub result: ExecutionResult,
    pub post_accounts: Vec<FixtureAccount>,
}
PLAIN ENGLISH

schema_version — Format version. Supports backward compatibility as Penrose evolves.

slot — The exact Solana block number where the event happened. Each block has a precise number.

recent_blockhash — A timestamp-like value that prevents transaction replay attacks.

signature — The transaction's unique ID. This is what you pass to delorean.

enabled_features — The list of feature flags active at that moment. Solana has hundreds of protocol toggles (called features) that activate over time. This field records which were active at that exact block.

lamports_per_signature — The cost per lamport fee at that moment. Fees change over time.

transaction — Raw transaction bytes: instructions, accounts, signatures.

pre_accounts — Every account state before the transaction: each wallet balance, token account, and program account as they stood before execution.

programs — The bytecode of every smart contract touched: each relevant program's bytecode (ELF binary), captured as it existed at that exact slot.

sysvars — System variables (like current rent price): Solana system variables — rent price, epoch info, clock. Crucial for accurate replay.

result — What the original validator recorded: status, consumed compute units, return data.

post_accounts — Every account state after the transaction: each account's state after execution. Used to verify replay accuracy.

With all 12 fields, you can rebuild the exact Solana universe that existed when that transaction ran — and replay it anywhere, anytime.

Why client diversity matters

When every validator client on Earth runs the same code, what happens when the code has a bug? In March 2023, an Agave validator bug caused a 4.5-hour outage on Solana mainnet. With only one client, the fix depended on a single team.

🌐
The resilience equation

Once Firedancer carries 30%+ of mainnet stake, a bug that only affects Agave might not stop the chain. Firedancer validators keep producing blocks while Agave gets patched. This is why Ethereum actively funds multiple client teams — client diversity is risk insurance at the infrastructure layer.

Client diversity also surfaces protocol ambiguities. When Firedancer and Agave disagree about a transaction's result, that's not a problem — it's a signal. One of them has a bug, and the disagreement tells you exactly which behavior to dig into.

Single-client network A single bug in the codebase can halt the whole chain. Every validator fails at the same time.
Two-client network (today) A bug in one client only affects validators running that client. The chain can keep producing blocks.
Three-client network (goal) Resilience multiplies. Byzantine fault tolerance significantly improves.

pmm-sim: The Private AMM Dissection Table

Same contract, same liquidity pool, same size — but different callers get different rates. pmm-sim lets you measure that gap locally, without sending a single mainnet transaction.

Scene

You use Jupiter to swap 15,000 USDC for WSOL. Your friend calls the same HumidiFi contract directly to do the same trade. She gets less WSOL.

Same contract, same pool, same size — why are the results different?

The answer lies in the design philosophy of private AMMs (Prop AMMs): they are not neutral. They identify the caller and offer better rates to whitelisted aggregators. pmm-sim lets you quantify that gap.

Solana's secret menu

Public AMMs (Raydium, Orca) price with public math, and everyone gets the same quote. Private AMMs are different — their pricing logic is closed-source, and they hand out different quotes based on the CPI caller's program address.

HumidiFi
Supports swap-v1 / v2 / v3 instructions across 5 WSOL-USDC markets, with significantly better quotes for Jupiter than for direct callers.
SolFi V2
Deep liquidity on WSOL-USDC and USDT-USDC markets, oracle-based pricing, and special treatment for DFlow.
GoonFi / BisonFi
Emerging private AMMs. Some restrict blacklisted addresses; whitelisted aggregators enjoy lower slippage.
Tessera / ObricV2 / ZeroFi
Focused on stablecoins or specific pairs. ObricV2 uses multi-oracle pricing; ZeroFi has a separate vault structure.
The four aggregators' "VIP passes"

pmm-sim can impersonate Jupiter (route_v2 instruction), DFlow (swap2 instruction), Titan (swap_route_v2 instruction), and OKX Labs (swap_v3_with_cpi_event instruction). Each aggregator has its own router program ID, and private AMMs use that ID to decide which quote tier to serve.

Two local SVMs: pmm-sim vs Delorean

pmm-sim and Delorean both run the SVM locally, but their starting points are completely different.

The Delorean way
// Pull a historical fixture from Penrose
let fixture = get_transaction_fixture(
    rpc, &signature
)?;
// Build a MockBank from historical state
let bank = build_mock_bank(&fixture);
// Replay that transaction
execute_transaction(&bank, &fixture.transaction);
The pmm-sim way

Delorean is an archaeologist — it restores the complete crime scene of a past transaction, every account, every sysvar straight from a historical snapshot, in order to replay and debug.

pmm-sim is a lab — it builds a clean LiteSVM environment from scratch, injects the accounts and programs you need, then runs swaps repeatedly to measure rates and compute usage. No historical baggage, reset at will, compare at will.

env.rs — environment init
let mut svm = LiteSVM::new()
    .with_default_programs()
    .with_sysvars()
    .with_sigverify(true)
    .with_compute_budget(budget);

// Inject token mints
mints.iter().for_each(|(mint, dec)| {
    svm.set_account(*mint,
        Misc::mk_mint_acc(*dec))
});

// Load programs (from disk or live from RPC)
svm.add_program_from_file(
    program_id, path_to_elf
);
In plain English

LiteSVM::new() — Spin up a blank in-process VM. No local validator needed; ready in milliseconds.

with_sysvars() — Inject Solana system variables (rent, clock, etc.), so programs think they're running on the real network.

svm.set_account() — Manually insert any account state. Can use a static snapshot cached on disk, or fetch the latest mainnet state via RPC (JIT mode).

add_program_from_file() — Load the private AMM's ELF bytecode. The program itself is publicly downloadable even when its logic is closed-source.

CPI spoofing: putting on Jupiter's coat

Private AMMs identify the caller by inspecting the program_id in the CPI chain. pmm-sim's core trick: replace Magnus Router's program_id with the target aggregator's program_id, so the private AMM thinks it's being called by Jupiter or DFlow.

builder.rs — instruction builder
pub fn instruction(
    &self,
    spoof: Option<Aggregator>,
    data: SwapArgs,
) -> Instruction {
    match spoof {
        None =>
            build_ix!(SwapBuilder),
        Some(Aggregator::Jupiter) =>
            build_ix!(RouteV2Builder),
        Some(Aggregator::DFlow) =>
            build_ix!(Swap2Builder),
        Some(Aggregator::Titan) =>
            build_ix!(SwapRouteV2Builder),
        Some(Aggregator::OkxLabs) =>
            build_ix!(SwapV3WithCpiEventBuilder),
    }
}
In plain English

spoof: None — Calls under Magnus Router's own identity; the private AMM prices you like a "regular source".

Aggregator::Jupiter → RouteV2Builder — Uses Jupiter's route_v2 instruction discriminator and substitutes Jupiter's address as program_id. The private AMM sees "Jupiter is calling me" and flips to the preferred-rate tier.

Each aggregator uses different instructions — This is key. Private AMMs don't just check program_id; they also check the instruction discriminator (first 8 bytes). Each aggregator's interface is slightly different, so a complete impersonation is required to trigger the better tier.

The payoff: run the same swap once with --spoof=jupiter and once with --spoof=dflow and you can quantify the rate gap between two aggregators inside the same private AMM.

PropAMMs go cross-chain: why EVM couldn't do this before

The mechanics pmm-sim quantifies — real-time quoting, identity recognition, private rates — are a uniquely Solana phenomenon. When the same logic meets Ethereum's infrastructure, it hits a wall.

Solana — why it works

No public mempool: transactions go straight to the leader, so no one can front-run a market maker's quote update.

Tiny per-transaction cost: each quote update is nearly free, so a market maker can update hundreds of times per second without losing money.

Jito priority lanes: high-value transactions can buy deterministic block ordering, making MEV predictable.

EVM — why it's hard

Public mempool = toxic flow: arbitrageurs see your quote-update transaction and can front-run it in the same block. Every update is an exposure.

Update cost is high: Ethereum gas makes high-frequency quote updates economically unviable.

State update latency: a 12-second block time means "real-time" quotes are actually historical data.

BaiBai: the first platform to bring PropAMMs to Base

In 2025, BaiBai (built on Spire Labs' Pylon) proposed a solution that makes PropAMM economics viable on EVM for the first time:

①
Dedicated block space for market makers
Quote-update transactions are guaranteed to land at the top of each block — market makers can update without being front-run inside the same block.
②
Speed bumps for toxic flow
Known arbitrage addresses get an automatic small delay (a speed bump), giving market makers a window to cancel stale quotes first.
③
Synchronous composability + smart routing
Composes natively with Base — no cross-chain bridge required. Large swaps not fully filled by BaiBai fall back automatically to public liquidity (external AMMs), avoiding liquidity fragmentation.
Why this matters to you

pmm-sim today only measures Solana's private AMMs. But the identity-sensitive pricing paradigm it quantifies is becoming the foundation of multi-chain DeFi. Understanding CPI spoofing, preferred-rate tiers, and benchmark data formats — these will show up again on BaiBai, and on future EVM PropAMMs.
