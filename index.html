什么是 Validator？

每一笔 Jupiter 兑换、每一次 NFT 铸造、每一次 DeFi 清算——所有这些都要经过一台叫做 validator 的机器。Validator 接收来自所有用户的全部 transactions， 对它们排序、执行，并将结果写入链上。 每 400 毫秒执行一次。

这 400ms 的窗口叫做 slot。 在每个 slot 中，一个被指定的领导者必须收集数千笔交易、验证每一个密码学签名、执行所有程序，并写出一个 block ——然后下一个 slot 就开始了。下面展示这段旅程的全貌：

Solana 的目标是每秒约 5,000 笔交易。这意味着 validator 必须大约每 0.2 毫秒处理一笔交易——比相机快门眨眼还快。这正是为什么 validator 工程与普通 Web 开发截然不同。

三大项目

Solana 最初只有一个 validator client： Agave（前身为 Solana Labs）。三支团队审视这一现状后，认为现有方案还不够好。

🔥

Firedancer

Jump Crypto 用 C 语言从零重建了整个 validator，目标是每秒 100 万笔交易——这是完整 Solana 协议的第二个独立实现。

🎵

Samba

Harmonic Finance 在 Firedancer 基础上分叉，并加入了 MEV 捆绑基础设施——为支付优先费用的交易者提供有保障的交易排序。

⏱️

Delorean

Temporal XYZ 构建了一项服务，将每笔交易捕获为可回放的快照，让你能够调试历史——精确回放任意一笔链上交易。

💡
为什么需要多个实现？

当两个独立代码库运行同一协议时，漏洞会以分歧的形式浮出水面，而不是悄无声息地破坏数据。工程师称之为 "consensus diversity" ——这是区块链所能拥有的最重要的安全属性之一。

02

Firedancer：C 语言打造的极速引擎

Jump Crypto 如何借鉴芯片制造工程理念、从零构建 validator：一个 CPU 核心，一项任务，零共享工具。

为什么选择 C？

Jump Crypto 使用与其 高频交易 系统相同的工程理念编写了 Firedancer： 一切用 C 编写，掌控每一个字节，绝不留有偶然。

⚡

无垃圾回收器

垃圾回收器 会在不可预知的时刻暂停程序执行。 在 HFT 场景下，哪怕一毫秒的暂停也意味着真实的金钱损失。 C 没有垃圾回收器——你精确掌控内存。

🎯

可预测的性能

Rust 提供内存安全保证，但其抽象层可能引入隐性开销。 C 编译后与你所写的代码一一对应——每次都达到纳秒级的可预测性。

🔍
独立押注

Firedancer 完全从零编写，与现有的 Agave validator 没有任何共享代码（Agave 用 Rust 编写）。这意味着 Agave 实现中的 bug 不会影响 Firedancer——反之亦然。 这就是"客户端多样性"的本质定义。

Tile：一个核心，一项任务

tile 是绑定到单个 CPU 核心 上的单线程，只做一件事。 NIC tile 负责接收数据包。 SigVerify tile 负责验证签名。 Pack tile 负责筛选哪些交易进入区块。 各 tile 之间从不直接共享内存——如同芯片制造流水线上的各个工位。

src/ Firedancer 源码根目录
tango/ 神经系统。所有 tile 通过环形缓冲区在此通信。
ballet/ 数学库。SHA-256、ed25519、base58——全部从零实现。
disco/ 通用流水线 tile：QUIC、SigVerify、Dedup、Pack。
discof/ 完整 Firedancer tile：共识、RPC、回放，无 Agave 依赖。
discoh/ Frankendancer tile：通往 Agave 执行层的桥接路径（混合模式）。
flamenco/ Solana 运行时。程序在 SVM 上实际执行的地方。
funk/ 账户数据库：支持分叉感知的内存键值存储。
waltz/ QUIC 之下的网络层——原始 UDP 与内核旁路。
wiredancer/ FPGA 加速模块——未来的硬件加速路径。
util/ C 环境基础设施：内存分配器、数据结构、数学原语。
💡
芯片制造工厂的比喻

把每个 tile 想象成半导体流水线上的专属工位。 没有工位会借用另一工位的工具，没有工位需要等待另一工位完成。 工件在传送带（即 环形缓冲区）上到来，完成处理，继续传递——速度与网线一致。

Tango：Tile 的神经系统

各 tile 之间从不通过相互调用来传递数据。 NIC tile 将原始交易字节写入共享内存区域，并更新 环形缓冲区 条目。 下游 tile 从同一块内存中读取数据——无需拷贝，无需 锁， 无需等待。 这是为纳秒级吞吐量而设计的 IPC。

🗂️

mcache

元数据缓存。 一个循环的"信封"环——每条条目描述一笔交易（其大小、在内存中的位置以及哈希值），而不包含数据本身。 写者在每个信封上盖章；读者扫描环形结构寻找新的印记。

📦

dcache

数据缓存。 实际的交易字节。一次性存储在共享内存中。 所有 tile 从同一位置读取——永远不会产生拷贝。 就像一块公告板，所有人都能阅读，无需各自复印一份。

🎟️

fseq

流序列器。 一个取号计数器。即使多个 NIC tile 同时写入，也能确保各 tile 按正确顺序读取消息。 就像熟食柜台的"当前服务号"显示屏。

🚦

fctl

流量控制。 一个背压信号。如果下游 tile 处理速度跟不上，fctl 会通知上游 tile 降速，而非溢出环形缓冲区。 就像高速公路拥堵时拦截来车的交通灯。

交易的旅程：400毫秒内发生了什么

从你的钱包点击到区块链最终确认 —— 400毫秒内经过六个检查点

400毫秒

你在 Jupiter 上点击兑换。在接下来的 400 毫秒内，你的交易在 Firedancer validator 内部经过六个独立检查点，随后被永久提交到 Solana 区块链。

400毫秒大约是打一个响指所需的时间。在这个窗口内，一个 validator 接收数千笔交易，对每一个签名进行密码学验证，检查重复项，为 区块 选出最优组合，执行每个程序，并将结果写入 支持分叉 的数据库。

1
QUIC 接入

原始 UDP 数据包通过 QUIC 从网络到达。NIC tile 将其解码为交易。

2
签名验证

每笔交易的 ed25519 签名 都会被检验。签名无效的交易将被静默丢弃。

3
Dedup 去重

64 位指纹用于识别 重复 提交。只有第一份副本才能继续传递。

4
区块打包

交易按 优先费 排名。Pack tile 为下一个区块选出最优的 2400 笔交易。

5
SVM 执行

SVM（Solana 虚拟机） 运行每个程序。 计算单元 被追踪并强制执行。

6
共识提交

区块 被分发给其他 validator。一旦 2/3 以上的节点达成共识，区块即被 最终确认 。

SVM 需要什么

在 SVM 能够执行哪怕一条指令之前，它必须加载一份完整的区块链状态快照。这段来自 Firedancer 测试框架的代码，展示了该快照所包含的确切内容。

CODE 

fn build_mock_bank(f: &TransactionFixture) -> MockBank {
    let bank = MockBank::default();
    {
        let mut accounts = bank.accounts.write().unwrap();

        for fa in &f.pre_accounts {
            accounts.insert(fa.pubkey,
                fixture_account_to_shared(fa));
        }

        for sv in f.sysvars.iter() {
            let mut acct = AccountSharedData::default();
            acct.set_data_from_slice(&sv.data);
            acct.set_owner(solana_sdk_ids::sysvar::id());
            acct.set_lamports(1);
            accounts.insert(sv.sysvar_id, acct);
        }
    }
    bank
}
通俗解释

创建一个空的内存"bank" —— 即 SVM 将要读取的 Solana 区块链状态模拟环境。

对于交易将涉及的每个账户：将其当前余额、所有者程序和存储数据加载到 bank 中。 公钥 用作查找索引。

加载所有 Solana 系统变量（ sysvars ）：当前租金价格、epoch 信息、最近 区块哈希 —— 以及运行时所需的 lamports 和所有权元数据。

返回填充完毕的 bank。SVM 现在拥有执行本笔交易所需的完整状态快照。

💡
零拷贝架构

在生产环境的 Firedancer 中，各 tile 之间从不复制交易数据。NIC tile 将交易字节一次性写入共享内存区域（dcache）。每个下游 tile 通过指针从同一内存位置读取数据。正因如此，Firedancer 能够每秒处理数十万笔交易 —— 内存带宽绝不浪费在数据复制上。

什么会导致交易失败

大多数失败的交易不会产生你所看到的错误 —— 它们只是从未到达目的地。了解是哪个检查点终止了交易，是调试的关键所在。

QUIC Ingest 数据包格式错误、网络格式不符，或该 validator 的 IP 被封锁。交易从未突破网络层。
SigVerify ed25519 签名 无效 —— 交易已损坏或使用了错误的密钥签名。立即丢弃，不收取手续费。
Dedup 这笔完全相同的交易已被见过 —— 是 validator 在数毫秒前已处理（或丢弃）的某笔交易的重发。钱包会不断重发；大多数重发都在此处被静默丢弃。
Pack slot 已满，或你的手续费太低 —— 该交易未被选入本次区块 slot。它可能在下一个 slot 中被重试。
Execute 程序耗尽了 计算单元 ，触及账户余额限制，或返回了错误代码。所有状态更改被回滚，但手续费仍然收取。

04

Samba — 网线速度下的 MEV

Harmonic Finance 如何修改 Firedancer，在验证节点层面实现有保证的交易排序

什么是 MEV？

交易排序中隐藏着利润。如果你能保证自己的 套利 交易恰好落在一笔大额价格冲击交换之后、其他人做出反应之前，那这个价差就归你所有。这就是 MEV — Maximum Extractable Value。

MEV 并非漏洞或攻击——它是任何由单一参与方（即 出块者 ）决定交易顺序的区块链上客观存在的经济现实。在 Solana 上，每个区块的生产 validator 决定哪些交易进入区块以及进入的顺序。

🔄

套利

在 DEX A 以低价买入代币，在同一区块内、价格均衡之前，在 DEX B 以高价卖出。

🥪

三明治攻击

在大额交换之前插入买单（ 抢先交易 ），等大额交换推动价格上涨后立即卖出。那笔大额交换就是三明治的"馅料"。

💧

清算

当借款人的抵押品价值跌破阈值时，争先成为第一个 清算人 并领取奖励。

📦

捆绑包

一组交易以 原子方式 保证按特定顺序落块——否则全部失败。用于安全地实施上述各类策略。

ℹ️
MEV 本身是中性的

MEV 可以提升市场效率（套利），也可以从普通用户身上提取价值（三明治攻击）。关键在于 validator 层面是否使这些保证成为可能。Samba 让这一切成为现实。

中文释义

v3.1.14 rebase = Harmonic 定期将最新的 Firedancer rebase 更新同步过来，使 Samba 始终跟上最新的性能改进。

harmonic buffer = Samba 新增了一个内部队列，在 bundle 到达 pack tile 之前先暂存进来。同时清理用于排序的 arrival_ns 时间戳。

reduce buffer time = 调整 bundle 在队列中等待的时长——等待时间过长会导致区块生产完毕前没来得及将其纳入。

pack: fix harmonic rejection condition = 精确定义何时拒绝一个 bundle：它能否放入剩余区块空间？目标 slot 是否距当前太远？

bundle: fix reset submit_leader_window_info_wait = bundle 只能在该 validator 即将成为 leader （出块者）时才能提交。此提交修复了该检查的时间窗口逻辑。

initial streaming impl = bundle 现在可以以实时流的方式提交，而非仅支持批量提交。这降低了每一毫秒都至关重要的高频交易者的延迟。

only append if not upcoming leader = 如果该节点即将不是出块者，则不再向队列追加 bundle——避免在无法使用的工作上浪费处理资源。

account for block txns in bundle/txn insert = 在选择 bundle 时，将区块中已选定的普通交易计入剩余空间的计算，确保数据准确。

为什么要修改 Validator？

为何不在应用层构建 MEV 工具——就像 Solana 上现有的解决方案那样？因为应用层的解决方案只能请求排序，只有 validator 本身才能保证排序。

💡
Validator 是唯一的真相来源

通过外部服务提交的 bundle，如果 validator 不明确遵守，仍可能被 抢先交易 。Samba 将 bundle 保证直接内嵌到 pack tile 本身，让排序由出块者在密码学层面予以保证。

以下是 Samba 与 Solana 上其他 MEV 排序方案的对比：

Samba bundle 在 validator 层面保证——pack tile 在调度任何其他交易之前先为 bundle 锁定位置
Jito（应用层） Validator 小费激励排序，但并非密码学锁定——恶意 validator 仍可重新排序
普通优先手续费 较高手续费可提高落块概率，但无法保证任何事——支付相同手续费的另一笔交易仍可能先行落块

Delorean — 交易时光机

Temporal 的 Penrose 服务如何在区块生成时捕获每笔交易的完整快照——并让你以外科手术般的精准重放历史

调试难题

🚨
场景描述

一个交易机器人在一笔本应成功的交易中损失了 40,000 美元。这笔交易深埋于 3 天前的某个 Solana 区块 中。网络已经向前推进。那个时刻的区块链状态在任何运行中的节点上都不再存在。你该如何调试？

标准的区块链调试几乎无从下手。你能看到交易失败了，但要理解失败的原因，你需要知道那个特定时刻每一个 账户、 每一个 program、 每一个系统变量的精确状态。而那个状态已经消失了。

Temporal XYZ 构建了 Penrose——一个与 Solana validator 并行运行的服务，在区块生成时为每笔交易捕获完整的状态快照。 Delorean 就是获取这些快照并重放它们的工具。

💡
这为何成为可能

Temporal 运营的基础设施在 validator 生产区块时捕获交易 fixture。 每个 fixture 包含 Solana VM 重放该交易所需的一切：交易前后的账户状态、programs、系统变量、已激活的功能标志。它是一个完美的时间胶囊。

什么是 Fixture？

每个 fixture 是一个包含 12 个字段的结构化记录——它们共同构成了一笔交易的完整 Solana 宇宙重建。

CODE 

pub struct TransactionFixture {
    pub schema_version: u32,
    pub slot: u64,
    pub recent_blockhash: Hash,
    pub signature: Signature,
    pub enabled_features: Vec<Pubkey>,
    pub lamports_per_signature: u64,
    pub transaction: Vec<u8>,
    pub pre_accounts: Vec<FixtureAccount>,
    pub programs: Vec<FixtureProgramData>,
    pub sysvars: Arc<Vec<FixtureSysvar>>,
    pub result: ExecutionResult,
    pub post_accounts: Vec<FixtureAccount>,

PLAIN ENGLISH

schema_version — 格式版本号。随着 Penrose 的演进，支持向后兼容。

slot — 事件发生时的精确 Solana 区块编号。 每个区块都有精确的编号。

recent_blockhash — 一个类似时间戳的值，用于防止交易重放攻击。

signature — 交易唯一ID。这正是你传给 delorean 的值。

enabled_features — 当时激活的功能标志列表。Solana 有数百个随时间激活的协议开关（称为 features）。此字段记录了该精确区块时哪些处于激活状态。

lamports_per_signature — 该时刻每个 lamport 的手续费成本。手续费会随时间变化。

transaction — 原始交易 字节： 指令、账户、签名。

pre_accounts — 交易前所有账户状态：每个钱包余额、代币账户和 program 账户在交易运行前的状态。

programs — 涉及的智能合约字节码：每个相关智能合约的 字节码（ELF 二进制）， 以该精确 slot 时的版本捕获。

sysvars — 系统变量（如当前 rent 价格）： Solana 系统变量—— rent 价格、epoch 信息、时钟。对于准确重放至关重要。

result — 原始 validator 记录的内容：状态、已使用的 计算单元、 返回数据。

post_accounts — 交易后所有账户状态：执行后每个账户的状态。用于验证重放准确性。

拥有全部 12 个字段，你就能重建那笔交易存在时的精确 Solana 宇宙——随时随地重放它。

🌐

单客户端网络 代码库中的一个漏洞就能导致整条链停止。所有 validator 同时失效。
双客户端网络（当前） 一个客户端的漏洞只影响运行该客户端的 validator。链可能继续生产区块。
三客户端网络（目标） 韧性成倍提升。 拜占庭容错 能力显著增强。

pmm-sim：私有 AMM 的解剖台

同一个合约、同一个流动性池、相同的金额——但不同身份的调用者会拿到不同的汇率。pmm-sim 让你在本地测量这个差距，一行 mainnet 交易都不需要发。

场景还原

你用 Jupiter 把 15,000 USDC 换成 WSOL。你的朋友直接调用同一个 HumidiFi 合约做同样的兑换。她拿到的 WSOL 更少。

同一个合约，同一个池子，同一笔金额——为什么结果不同？

答案在于 私有 AMM（Prop AMM）的设计哲学：它们不是中立的。它们识别调用者的身份，并为白名单聚合器提供优惠汇率。pmm-sim 让你量化这个差距。

Solana 的秘密菜单

公开的 AMM（如 Raydium、Orca）用公开的数学公式定价，任何人得到相同的价格。私有 AMM 则不同——它们的定价逻辑是闭源的，并且会根据 CPI 调用者的程序地址给出不同的报价。

HumidiFi
支持 swap-v1 / v2 / v3 三种指令，5 个 WSOL-USDC 市场，对 Jupiter 的报价显著优于直接调用。
SolFi V2
流动性深厚的 WSOL-USDC 和 USDT-USDC 市场，使用预言机定价，对 DFlow 有特别优惠。
GoonFi / BisonFi
新兴私有 AMM，部分对黑名单地址有限制，白名单聚合器享受更低滑点。
Tessera / ObricV2 / ZeroFi
专注于稳定币或特定交易对，ObricV2 使用多路预言机，ZeroFi 有独立的 vault 结构。
四大聚合器的"VIP 通行证"

pmm-sim 支持伪装成 Jupiter（route_v2 指令）、DFlow（swap2 指令）、Titan（swap_route_v2 指令）、OKX Labs（swap_v3_with_cpi_event 指令）。不同聚合器对应不同的路由程序 ID，私有 AMM 通过识别这个 ID 来决定给哪档报价。

两种本地 SVM：pmm-sim vs Delorean

pmm-sim 和 Delorean 都在本地运行 SVM，但它们的出发点截然不同。

Delorean 方式
// 从 Penrose 拉取历史 fixture
let fixture = get_transaction_fixture(
    rpc, &signature
)?;
// 用历史状态构建 MockBank
let bank = build_mock_bank(&fixture);
// 重放那笔交易
execute_transaction(&bank, &fixture.transaction);
pmm-sim 方式

Delorean 是考古学家——还原过去某笔真实交易的完整现场，每个账户状态、每个系统变量都来自历史快照，目的是重放和调试。

pmm-sim 是实验室——从零搭建一个干净的 LiteSVM 环境，注入你需要的账户和程序，然后反复执行 swap，测量汇率和算力消耗。没有历史包袱，随时重置，随时对比。

env.rs — 环境初始化
let mut svm = LiteSVM::new()
    .with_default_programs()
    .with_sysvars()
    .with_sigverify(true)
    .with_compute_budget(budget);

// 注入代币 mint
mints.iter().for_each(|(mint, dec)| {
    svm.set_account(*mint,
        Misc::mk_mint_acc(*dec))

// 加载程序（从磁盘或 RPC 实时拉取）
svm.add_program_from_file(
    program_id, path_to_elf

中文释义

LiteSVM::new() — 创建一个空白的进程内虚拟机，不需要启动本地验证节点，毫秒级就绪。

with_sysvars() — 注入 Solana 系统变量（rent、clock 等），让程序以为自己在真实网络上运行。

svm.set_account() — 手动插入任何账户状态。可以用磁盘缓存的静态快照，也可以通过 RPC 实时拉取 mainnet 最新状态（JIT 模式）。

add_program_from_file() — 加载私有 AMM 的 ELF 字节码。程序本身是公开可下载的，即使逻辑闭源。

CPI 伪装：穿上 Jupiter 的外衣

私有 AMM 通过检查 CPI 调用链里的 program_id 来判断调用者身份。pmm-sim 的核心技巧：把 Magnus Router 的 program_id 替换成目标聚合器的 program_id，让私有 AMM 以为是 Jupiter 或 DFlow 在调用它。

中文释义

spoof: None — 用 Magnus Router 本身的身份调用，私有 AMM 按"普通来源"定价。

Aggregator::Jupiter → RouteV2Builder — 使用 Jupiter 的 route_v2 指令鉴别符（discriminator），并把 program_id 替换成 Jupiter 的地址。私有 AMM 看到"Jupiter 在调用我"，切换到优惠汇率档。

每个聚合器对应不同指令 — 这是关键。私有 AMM 不只是检查 program_id，它们还检查指令的鉴别符（前 8 字节）。每个聚合器的接口略有不同，必须完整仿冒才能触发优惠档位。

结果：同一笔 swap，用 --spoof=jupiter 和 --spoof=dflow 各跑一次，就能量化两个聚合器在同一个私有 AMM 里的汇率差距。

PropAMM 出海：为什么 EVM 之前做不到

pmm-sim 所量化的这套机制——实时报价、身份识别、私有汇率——是 Solana 特有的产物。当同样的逻辑遇到以太坊的基础设施，会撞上一堵墙。

Solana — 为什么能

无公开 mempool：交易直接发给领导者，做市商更新报价前没有人能抢跑。

微交易成本极低：每笔报价更新几乎免费，每秒可更新数百次而不亏损。

Jito 优先通道：高价值交易可以买到确定性的区块排序，MEV 可预测。

EVM — 为什么难

公开 mempool = 毒流量：套利者看到你的报价更新交易，可以在同一区块里抢先执行。每次更新都是暴露。

更新成本高：以太坊 gas 费让高频报价更新经济上不可行。

状态更新延迟：12 秒出块周期让"实时"报价实为历史数据。

BaiBai：第一个把 PropAMM 带到 Base 的平台

2025 年，BaiBai（基于 Spire Labs 的 Pylon 构建）提出了一套破局方案，让 PropAMM 的经济模型在 EVM 上首次成立：

①
做市商专属区块空间
报价更新交易保证排在每个区块最顶部——做市商更新报价时，不会被套利者在同一区块内抢跑。
②
对毒流量设置速度限制
检测到已知套利地址后，自动添加小延迟（speed bump），让做市商有窗口先撤销旧报价。
③
同步可组合性 + 智能路由
与 Base 原生同步组合，无需跨链桥。未被 BaiBai 填满的大额 swap 自动回退到公开流动性（外部 AMM），避免流动性碎片化。
为什么这对你重要

pmm-sim 目前只测量 Solana 上的私有 AMM。但它所量化的身份敏感定价范式，正在成为多链 DeFi 的基础设施。理解 CPI 伪装、优惠档汇率、基准数据格式——这些知识在 BaiBai、在未来的 EVM PropAMM 上都会再次出现。
